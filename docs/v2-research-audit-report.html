1. MediaTek NeuroPilot SDK 对比

1.1 工具链名称与CLI

项目之前（错误）现在（基于真实文档）
转换器名称 np-converter（泛称） NP Converter NP8 官方名称；Public版仅NP8可用
编译器名称 ncc-tflite（笼统） Neuron Compiler (ncc-tflite) → 生成 .dla 文件
是 Neuron SDK 的一部分，位于 SDK bundle 的 SoC 专属目录下
运行时 neuron-runtime（未区分 CLI 和 API） neuronrt（CLI工具）+ Neuron Runtime API（C++ API）
两个不同组件，用途不同
Converter 参数 未指定具体CLI参数 --tflite_op_export_spec np='sdk_v6' 可生成NP6/NP7兼容模型
Genio平台仅支持NP8 Converter

1.2 平台映射关系

平台SoC代号NP版本MDLA版本OS支持SDK获取方式
Genio-720 mt8189 NP8 MDLA 5.3 Android + Yocto 需单独下载 SDK All-In-One Bundle 不包含 mt8189
Genio-520 mt8189 NP8 MDLA 5.3 Android + Yocto 同上
Genio-510 mt8390 NP6 — Android + Yocto SDK bundle
Dimensity 9400 mt6991 NP8 MDLA 5.3 Android NDA Basic/Premium
Dimensity 9300 mt6989 NP7/NP8 MDLA 5.2 Android NDA Basic/Premium
❌ 关键错误：SoC代号缺失
之前的 platform_profile 缺少 SoC 代号（mt6991, mt6989, mt8189），而 SDK bundle 的目录结构按代号组织。
修复： 在 profile 中添加 soc_codename 字段，并记录 SDK 下载路径规则。
❌ 关键错误：Genio-720 SDK 不在 All-In-One Bundle 中
Genio-720/520 对应 mt8189，但 NeuroPilot SDK All-In-One Bundle 不包含 mt8189 二进制文件。
需要从 NeuroPilot 门户单独下载 mt8189 包。
修复： 在 profile 中记录 sdk_download_note 字段。

2. Qualcomm QNN SDK 对比

2.1 工具链名称与CLI

项目之前（错误）现在（基于真实文档）
转换器名称 qnn-converter（不存在） qnn-pytorch-converter (TorchScript → QNN)
qnn-onnx-converter (ONNX → QNN)
两个不同的转换器
PyTorch 转换 未指定参数 必须指定 --input_dim：
qnn-pytorch-converter --input_network model.pt --input_dim "input" 1,3,299,299 --output_path model.cpp
ONNX 转换 未指定参数 支持 --dry_run 分析不支持的算子
支持 ONNX Opset ≤ 24
自动运行 onnx-simplifier
编译器名称 qnn-context-binary-generator（名称正确） qnn-context-binary-generator ✓
参数：--backend libQnnHtp.so --model model.so --config_file config.json --binary_file output.serialized
运行时后端 "qnn" 泛称 具体后端库：
libQnnHtp.so (HTP/NPU)
libQnnCpu.so (CPU)
libQnnGpu.so (GPU)

之前的 vendor_extensions 只记录了简单的 key-value，没有这个结构。

4.2 关键代码问题

1. 转换器名称错误传播到多个文件
catalog.py 中 DEFAULT_VENDOR_ADAPTERS 和 DEFAULT_TASK_TEMPLATES 使用了错误的工具名称：
• "converter": "qnn-converter" → 应为 qnn-pytorch-converter / qnn-onnx-converter
• "compiler": "qnn-context-binary-generator" → 名称正确，但参数描述不准确

DEFAULT_PLATFORM_PROFILES 中 "converter": "qnn-converter" 同样错误。
2. MediaTek 工具链描述过于简化
DEFAULT_VENDOR_ADAPTERS["mediatek-neuropilot"] 的 conversion_pipeline：
• {"step": "convert", "tool": "np-converter", ...} — 工具名应为 "NP Converter"
• {"step": "quantize", "tool": "ptq-pipeline", ...} — 不存在 "ptq-pipeline" 工具，PTQ 是 NP Converter 的内置功能
• {"step": "compile", "tool": "ncc-tflite", ...} — 名称正确但需注明是 SoC 专属版本
3. SoC codename 缺失
DEFAULT_PLATFORM_PROFILES 中所有平台都缺少 soc_codename 字段。
真实映射：Dimensity 9400 = mt6991, Dimensity 9300 = mt6989, Genio-720 = mt8189。
这个字段对于 SDK 二进制文件定位至关重要。
4. QNN HTP 性能模式未集成
DEFAULT_VENDOR_ADAPTERS["qualcomm-qnn"] 和 DEFAULT_PLATFORM_PROFILES["qualcomm-snapdragon-*"] 都未记录 HTP 性能模式。
真实场景中，选择正确的性能模式（burst vs balanced vs power_saver）对推理性能影响巨大。
5. NDA 访问级别未记录
MediaTek NeuroPilot 有 Public/Basic/Premium 三个层级，不同层级可用的量化方案和工具不同。
当前 platform_profile 未区分，可能导致用户尝试使用需要 NDA 的功能。
6. device_farm.py 的 _simulate_device_result 缺少 Qualcomm 延迟偏差
_simulate_device_result 中有 dimensity 和 kirin 的延迟偏差，但缺少 snapdragon 的偏差。
真实场景中，Snapdragon 8 Gen 3 的 NPU 性能通常优于 Dimensity 9300。
7. intent_planner 中 vendor_hint 检测可改进
当前检测关键词较基础。可以添加：
• "ncc", "neuron", "dla" → MediaTek
• "hexagon", "htp", "qnn" → Qualcomm
• SoC 型号关键词（mt6991, sm8650 等）
8. Skill 文档缺少真实 CLI 示例
platform-conversion-mediatek/SKILL.md 和 platform-conversion-qualcomm/SKILL.md 描述了工作流但缺少真实的命令行示例。
应添加如：
NP Converter --input_network model.pt --output_path model.tflite --tflite_op_export_spec np='sdk_v6'
qnn-pytorch-converter --input_network model.pt --input_dim "input" 1,3,224,224 --output_path model.cpp
